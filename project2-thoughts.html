Project 2: Audio Clipper Thoughts

Project 2: Audio Clipper Thoughts

This project is going pretty well. It works and my friends and I have actually used it already. There might be a few issues with the design, though. I tried to go for a very minimal, somewhat suckless inspired style. Although I still enjoy the result, it feels like I've just purposefully left out some desirable features. I'll specify more in section two.

I should also note that I found an existing program that aims to accomplish my goal with this project almost exactly. That project is called timemachine. When I tried to use it, it didn't work exactly like I expected. Although I probably missed the feature for retroactive recording, I still decided to go ahead with this project.

1 JACK API

Getting started with the JACK API was extremely easy. There is plenty of documentation and example code provided. The simple_client.c example was a helpful starting point for this project. While looking at the source for timemachine, I also saw some distinct leftovers from simple_client.c there.

From that simple client, I just needed to make the number of ports adjustable.

2 Sound Encoding and Container Formats

It was pretty quick to set up the ring buffers and write the contents from the input ports to their corresponding buffers. However, it took much more time to decide on how that data would eventually make it to the disk.

I looked at the library used in timemachine, libsndfile, but thought there were better alternatives. I didn't find any recommendations for sound libraries from suckless.org, but I saw that the tools they recommended supported formats like flac and vorbis from xiph.org. Turns out that xiph.org has made many really excellent and open sound encoding and container formats.

So I tried first to use libflac and liboggz, then libfishsound and liboggz, and maybe some other permutations of their library offerings. I couldn't exactly figure them out, which was disappointing. However, I came to the conclusion that instead of adding that code to my program, I could just use the existing code in ffmpeg. That tool already supports encoding from raw little-endian 32-bit floating point pcm, so I now just write the raw audio to the disk and then use ffmpeg to convert it to any other format imaginable. I think this is extremely in-line with suckless thinking, especially because their video editor, blind, just deals with raw, uncompressed video.

This might be a little off putting though. I may end up changing this later. It would also be a good idea to learn how to deal with these formats properly anyway.

3 Signal Handling

We recently completed a project involving signal handling in my Systems 2 course at the Ohio State University. So I looked to that project for implementing a method in which sending a signal to the process would write the contents of the buffer to the disk. It doesn't completely work though, because the program will also exit after writing when receiving the signal.

4 Misc

I was thinking these midpoint project reviews would make for a good opportunity to cover the miscellaneous things I read and learned from during development.

Early on, I found the Linux kernel style guide and skimmed a few sections. It was mentioned early on that most of the direction was taken from K&R. I was surprised I didn't make the choice to follow the example from that book already. Seems pretty obvious now. Now I have a clear idea on how to style my C code. The style guide also provided some Emacs configuration for making some of the formatting automatic, which is nice.

I also was in the habit of following some of the recent work of the giants from Bell Labs this month. I was reminded how eloquent Brian Kerninghan is and found his talk on "The Elements of Programming Style". In the introduction, he discussed the book and also its basis in "The Elements of Style" by Strunk and White. The book seemed familiar. After looking around, I found my Dad had two copies of the book. I've already been writing these blog posts to get some practice with my speaking and writing, so I'm excited to read this book and hopefully get closer to Kerninghan's skill level.

I'll be back in another two weeks.
